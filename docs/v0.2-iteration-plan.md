# v0.2 Iteration Plan

## Product Slice

Target user: individual developers localizing their own software, games, or
media projects.

Primary outcome: generate target-locale artifacts that can be dropped into the
project in the correct location. Localize Anything does not repackage APKs,
sign binaries, or automatically rewrite application code.

First agent shape: a protocol-backed localization agent. The protocol defines
portable artifacts and deterministic safety gates; the agent routes project
text, prepares parallel LLM translation work, imports host LLM responses, and
reflects through review, QA, dashboard, and apply-plan outputs before the user
decides whether to overwrite project files.

## Agent Granularity Decisions

Decision date: 2026-06-20.

The next iteration optimizes for a complete agent architecture rather than a
single polished benchmark demo or Web UI pass. The target shape is still a
local developer tool, but the runtime should behave like an agent system with
clear internal ownership boundaries:

- Project Agent: owns project session state, ignore rules, source selection,
  text-type routing, benchmark configuration, and evidence for why files were
  included or excluded.
- Generation Agent: owns batch planning, direct provider execution, manual
  handoff fallback, response import, validation, and failed-batch retry.
- Review Agent: owns deterministic QA, LLM reflection/critique, review-sheet
  output, and classification of issues that require developer or translator
  judgment.
- Delivery Agent: owns staged copies, delivery packaging, dashboards, apply
  plans, and explicit run-id-confirmed overwrite with backups.
- Web UI: remains a control panel and review surface over the shared runtime
  APIs. It must not become the place where workflow logic lives.

Selected scope decisions:

- Success standard: architecture-complete agent system.
- LLM integration: direct provider API plus provider-agnostic handoff/import
  fallback. Both paths must produce the same generated segment JSONL contract.
- Review depth: deterministic QA plus LLM reflection plus human-readable review
  sheets.
- First real-world target: Android open-source source projects.
- Automation boundary: write staged copies first; overwrite the source project
  only after explicit user confirmation.
- Run state: keep artifact-first run directories, and add a session index so
  the CLI/Web UI can resume prior work and rerun failed batches.

Pressure-test criteria for this slice:

- Project routing fails the slice if generated outputs, old `localize-run-*`
  folders, package directories, or backup files are selected as source inputs in
  a normal open-source project scan.
- Generation fails the slice if a failed batch cannot be rerun or imported
  without rerunning the whole project.
- Direct provider integration fails the slice if it bypasses or weakens the
  handoff/import validation contract.
- Review fails the slice if LLM reflection is only a summary and does not emit
  actionable segment-level issues that can block or annotate delivery.
- Delivery fails the slice if any command overwrites source project files
  without staged output, apply-plan review, and run-id confirmation.

Implemented Project Agent baseline:

- `inspect_project` applies a shared scan policy that ignores generated
  `localize-run-*` folders, `.localize-anything`, default output folders,
  dependency/build directories, and common compiled artifacts.
- Project inspection emits routing evidence through `scan_policy`,
  `ignored_paths`, and `skipped_paths` so source selection can be audited.
- `localize-run` and `agent-run` record resumable sessions in
  `.localize-anything/sessions/index.json`.
- `agent-summary.json` now includes scan evidence and a pointer to the session
  index.
- The CLI exposes `sessions`; the Web UI backend exposes `/api/sessions`.
- Protocol validation includes `project-session` and `agent-summary` schemas
  and examples.

Implemented Generation Agent baseline:

- `provider-generate` can execute a generation handoff through a direct HTTP
  JSON provider. The provider response is normalized with the same
  `import-generated-response` contract used by manual handoff/import.
- `agent-run --provider-url` runs the same direct provider path and then
  continues into delivery packaging when provider generation validates.
- `retry-handoff` creates a smaller generation handoff from failed import or
  collect reports so missing/broken batches can be rerun without re-extracting
  the whole project.
- Retry handoffs reuse the standard `generation-handoff` schema and generated
  segment JSONL contract.

Implemented Review Agent baseline:

- Successful generated runs now emit `llm-review-request.json` and
  `llm-review-prompt.md` beside the human review sheet.
- `llm-review-request` can create provider-agnostic reflection prompts from any
  generated segment JSONL file.
- `import-llm-review` normalizes LLM reviewer responses into segment-level
  issues with severity, category, confidence, and optional suggested target.
- Protocol validation includes `llm-review-request` and `llm-review-result`
  schemas and examples.

Implemented Delivery Agent baseline:

- Successful delivery runs now emit `delivery-decision.json` and
  `delivery-decision.md` beside the dashboard and apply plan.
- `delivery-decision` classifies QA findings, apply operations, conflicts, and
  unprocessed assets as `blocked`, `owner_review_required`, or
  `ready_no_changes`.
- `agent-summary.json` now exposes a first-class `delivery` section so
  orchestration can see whether user confirmation, developer review, or conflict
  resolution is required.
- Protocol validation includes the `delivery-decision` schema and example.

## v0.2.1 Reference Policy Baseline

Implemented on 2026-06-20.

The runtime now treats localization context as an explicit protocol contract
instead of an implicit project assumption:

- `operating_mode` is first-class in project config, batch plans, work packets,
  draft requests, delivery manifests, project sessions, and agent summaries.
- `reference_policy` is first-class across the same protocol surfaces.
- Supported operating modes are `blind_benchmark`, `greenfield_localization`,
  `existing_locale_maintenance`, and `rewrite_or_harmonization`.
- Supported reference policies are `blind`, `style_only`, `tm_assisted`, and
  `preserve_existing`.
- Default greenfield behavior is `style_only`: approved terminology and style
  guidance can be used, but existing target segment translations are not
  assumed to be truth.
- Blind benchmark behavior is `blind`: target references, glossary targets, and
  translation memory targets are excluded from generation work packets and
  draft requests. Reference material is only visible in evaluation artifacts.
- Existing locale maintenance defaults to `preserve_existing`: reviewed TM
  records whose source hash is unchanged are preserved outside generation,
  while missing, stale, conflicting, or unreviewed segments remain candidates.
- Existing target references are reported in `reference-plan.json`, including
  obsolete target references, but the runtime does not delete target files.

Verification added:

- `LocalizeRunTests.test_blind_benchmark_hides_target_references_from_generation_packets`
  proves blind benchmark packets and draft requests do not leak approved target
  strings from glossary or translation memory.
- `LocalizeRunTests.test_existing_locale_maintenance_preserves_reviewed_unchanged_segments`
  proves reviewed unchanged translations are excluded from generation batches
  and restored into the staged target language file.
- `ProjectTests.test_plan_and_retrieve_context_packet` now explicitly requests
  `rewrite_or_harmonization` to test the TM-assisted retrieval path, preserving
  the new greenfield default.

## Platform Slices

### Android Source Projects

The Android platform slice supports source projects with
`res/values/strings.xml`.

Workflow:

```text
detect android strings source
-> extract string, string-array, and plural resources
-> build work packets for LLM draft translation
-> create provider-agnostic draft requests
-> create host-agent generation handoff manifest
-> normalize host-agent LLM responses when needed
-> collect and validate generated segment JSONL
-> route generated segments with stage-generated
-> write staged res/values-<locale>/strings.xml
-> run deterministic Android string QA
-> package delivery snapshot
-> render developer/translator delivery dashboard
-> render Delivery Agent decision report
-> render non-mutating apply plan for user approval
-> optional run-id-confirmed apply with backups
```

Out of scope for the first slice:

- APK decompilation, signing, or repackaging.
- Automatic Gradle or project-code edits.
- Layout, drawable, audio, image, and binary resource localization.
- Inline markup and Android resource forms beyond strings, arrays, and plurals.
  These are reported for developer follow-up.

### iOS Source Projects

The iOS platform slice supports source projects with `.lproj` `.strings` and
`.stringsdict` resources, plus Xcode `.xcstrings` String Catalogs.

Workflow:

```text
detect iOS .lproj source
-> extract .strings key/value and .stringsdict plural resources
-> build work packets for LLM draft translation
-> create provider-agnostic draft requests
-> create host-agent generation handoff manifest
-> normalize host-agent LLM responses when needed
-> collect and validate generated segment JSONL
-> route generated segments with stage-generated
-> write staged <locale>.lproj resource files
-> run deterministic iOS string QA
-> package delivery snapshot
-> render developer/translator delivery dashboard
-> render Delivery Agent decision report
-> render non-mutating apply plan for user approval
-> optional run-id-confirmed apply with backups
```

String Catalog workflow:

```text
detect .xcstrings source
-> extract source-language stringUnit and variation leaf resources
-> build work packets for LLM draft translation
-> create provider-agnostic draft requests
-> create host-agent generation handoff manifest
-> normalize host-agent LLM responses when needed
-> collect and validate generated segment JSONL
-> route generated segments with stage-generated
-> write staged catalog copy with target language entries
-> run deterministic String Catalog QA
-> package delivery snapshot
-> render developer/translator delivery dashboard
-> render Delivery Agent decision report
-> render non-mutating apply plan for user approval
-> optional run-id-confirmed apply with backups
```

Out of scope for the iOS slice:

- IPA extraction, signing, or repackaging.
- Automatic Xcode project, Info.plist, Swift, or Objective-C edits.
- Storyboard, asset catalog, image, audio, and binary resource localization.
- String Catalog variation forms that cannot be represented as stringUnit
  leaves.

## LLM Generation Boundary

LLM translation generation is the core workflow stage, but the deterministic
runtime remains provider-agnostic. The host agent consumes work packets and
produces segment JSONL with `target`, `target_locale`, and generation metadata.
The runtime validates, stages, packages, and applies the resulting artifacts.

Runtime commands for this boundary:

- `ui`: start a local web workbench bound to localhost. The interface calls the
  same protocol-backed runtime APIs for project intake, routing review, handoff
  creation, response import, review-sheet preview, dashboard preview, and
  apply-plan inspection.
- `agent-run`: first provider-agnostic agent shell. It records routing
  decisions, creates parallel handoff prompts, optionally imports response
  directories or calls a direct HTTP provider, and runs reflection through
  review, deterministic QA, delivery dashboard, and non-mutating apply-plan
  artifacts.
- `android-app-test`: run the Android agent flow against an isolated app copy,
  apply the generated language resource to that copy with run-id confirmation,
  and revalidate Android resources after apply while proving the original
  project was not mutated. It defaults to synthetic draft for engineering
  verification, accepts `--generated` or `--generated-dir` for Codex/host
  generated Chinese drafts, and can generate a local Chinese draft internally
  with `--local-chinese-draft`. Use `--require-real-generation` to fail unless
  non-synthetic generated text with a non-empty quality claim is supplied.
- `generate-chinese-draft`: create conservative local Chinese draft generated
  segment JSONL from extracted segments for app-copy E2E testing before a real
  provider is wired. It marks output as `provider: codex-local` and
  `quality_claim: local_chinese_draft_for_e2e`; it is not translation-quality
  evidence.
- `sessions`: list the artifact-first project session index written by
  `localize-run` and `agent-run`.
- `localize-run`: orchestrate preflight, extraction, batching, draft-request
  emission, generated-batch collection, staging, deterministic QA, delivery
  packaging, and dashboard creation for one target locale.
- `deepseek-generate`: translate extracted segment JSONL via DeepSeek API with
  automatic placeholder parity. First direct LLM provider integration, proven
  with AntennaPod en-US→ja+ko (869 segments, QA pass). See
  `docs/test-antennapod-deepseek.md`.
- `draft-request`: turn one work packet into host-agent instructions.
- `render-draft-prompt` / `render-handoff-prompts`: render provider-agnostic
  draft requests as paste-ready Markdown prompts for manual host-agent
  generation.
- `import-generated-response`: normalize common LLM response shapes into
  canonical generated segment JSONL while preserving source fields from the work
  packet.
- `import-generated-handoff`: normalize a response directory for every batch in
  a handoff and write the expected generated JSONL batch files.
- `retry-handoff`: create a new handoff containing only failed or missing
  batches from an import/collect report.
- `provider-generate`: run a generation handoff through a direct HTTP JSON
  provider while keeping the same generated segment JSONL validation contract.
- `generation-handoff`: list all draft requests and expected generated JSONL
  paths for a run.
- `collect-generated`: validate every generated batch against its work packet
  and combine them into one generated segment JSONL file.
- `validate-generated`: validate an individual generated batch.
- `review-sheet`: export generated segment JSONL as Markdown and CSV so
  developers or translators can inspect draft translations before apply.
- `llm-review-request` / `import-llm-review`: create provider-agnostic LLM
  reflection prompts and normalize returned segment-level review issues.
- `stage-generated`: route combined generated segment JSONL to the correct
  adapter and write staged target files across mixed project formats.
- `delivery-decision`: render an explicit Delivery Agent decision report from
  staged delivery and apply-plan evidence.
- `plan-apply --markdown-output`: render a non-mutating apply plan so the user
  can approve exact create/replace/conflict operations before overwriting files.

## Safety Model

All localized files are written to staging first. Applying to the original
project requires explicit confirmation by delivery `run_id`. Replacement writes
create backups under `.localize-anything/backups/<run_id>/`.

For source software projects, the default product promise is language-resource
delivery, not binary repackaging. If a project needs code, build-system, layout,
or asset changes to consume a localized resource, the runtime should report that
as a developer decision instead of applying automatic source edits.
